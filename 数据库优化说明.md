# 停车数据查询优化方案

## 📊 优化目标

- **查询速度提升**: 5-50倍（取决于数据量）
- **并发能力提升**: 10-20倍
- **响应时间**: 从2-5秒降低到0.1-0.5秒

---

## 🎯 优化方案一：数据库索引优化 ⭐⭐⭐⭐⭐

### 问题分析

当前停车数据查询使用4个表：
- `tnxz_dx_vehicle_province_rank_byday` (天女小镇-按省份)
- `tnxz_dx_vehicle_city_rank_byday` (天女小镇-按城市)
- `hlw_vehicle_province_rank_byday` (欢乐湾-按省份)
- `hlw_vehicle_city_rank_byday` (欢乐湾-按城市)

**查询特点：**
```sql
SELECT a.location, SUM(a.vehicle_count) AS vehicle_count
FROM table_name a
WHERE a.`出场日期` BETWEEN '2025-05-01' AND '2025-05-05'
GROUP BY a.location
ORDER BY vehicle_count DESC
LIMIT 10;
```

**性能瓶颈：**
- ❌ `出场日期` 字段没有索引 → 全表扫描
- ❌ 日期范围查询慢
- ❌ GROUP BY 需要临时表

### 实施步骤

#### 步骤1：执行索引创建脚本

```bash
# 连接到MySQL数据库
mysql -u your_user -p carpark

# 执行优化脚本
source database_optimization_parking.sql
```

或者使用MySQL客户端工具（如Navicat、DBeaver）执行 `database_optimization_parking.sql`

#### 步骤2：验证索引创建

```sql
-- 查看索引
SHOW INDEX FROM tnxz_dx_vehicle_province_rank_byday;

-- 查看执行计划
EXPLAIN SELECT a.location, SUM(a.vehicle_count) AS vehicle_count
FROM tnxz_dx_vehicle_province_rank_byday a
WHERE a.`出场日期` BETWEEN '2025-05-01' AND '2025-05-05'
GROUP BY a.location
ORDER BY vehicle_count DESC
LIMIT 10;
```

**优化前的执行计划：**
```
type: ALL          # 全表扫描
rows: 1000000      # 扫描100万行
Extra: Using temporary; Using filesort
```

**优化后的执行计划：**
```
type: range        # 索引范围扫描
key: idx_tnxz_province_date
rows: 5000         # 只扫描5000行（减少200倍）
Extra: Using where; Using index
```

### 索引说明

为每个表创建了2个索引：

1. **复合索引** `idx_xxx_date_location`
   - 字段：`出场日期` + `location`
   - 用途：同时优化WHERE和GROUP BY
   - 适用场景：大部分查询

2. **单列索引** `idx_xxx_date`
   - 字段：`出场日期`
   - 用途：优化日期范围查询
   - 适用场景：日期筛选

### 预期效果

| 数据量 | 优化前 | 优化后 | 提升倍数 |
|--------|--------|--------|---------|
| 10万行 | 1-2秒 | 0.1-0.2秒 | 10倍 |
| 100万行 | 3-5秒 | 0.2-0.5秒 | 15倍 |
| 1000万行 | 10-20秒 | 0.5-1秒 | 20-40倍 |

## 🎯 优化方案二：数据库连接池 ⭐⭐⭐⭐

### 问题分析

**当前问题：**
- 每次查询都创建新的数据库连接
- 连接建立时间：50-200ms
- 高并发时连接数耗尽

**优化方案：**
使用DBUtils连接池，复用数据库连接

### 实施步骤

#### 步骤1：安装依赖

```bash
pip install DBUtils
```

#### 步骤2：更新requirements_web.txt

已创建新的依赖文件，包含：
- `DBUtils>=3.0.0` - 数据库连接池

#### 步骤3：使用连接池（可选）

新创建的 `models/database_pool.py` 提供了连接池实现。

**如果要启用连接池，需要修改现有的Model类：**

```python
# 从
from .database import DatabaseModel

# 改为
from .database_pool import DatabaseModelWithPool as DatabaseModel
```

### 连接池配置说明

```python
maxconnections=20,    # 最大连接数（根据并发用户调整）
mincached=2,          # 初始空闲连接数
maxcached=10,         # 最大空闲连接数
blocking=True,        # 连接池满时等待
ping=1,               # 自动检测连接有效性
```

### 预期效果

**连接建立时间：**
- 优化前：每次50-200ms
- 优化后：复用连接，0ms

**并发能力：**
- 优化前：频繁创建/销毁连接，最多5-10并发
- 优化后：连接池复用，支持20-50并发

**总体提升：**
- 响应时间减少：10-20%
- 并发能力提升：10-20倍
- 数据库压力减少：50-70%

## ⚠️ 注意事项

### 索引维护

1. **定期分析表**
```sql
ANALYZE TABLE tnxz_dx_vehicle_province_rank_byday;
```

2. **监控索引使用情况**
```sql
-- 查看索引统计
SHOW INDEX FROM tnxz_dx_vehicle_province_rank_byday;

-- 查看未使用的索引
SELECT * FROM sys.schema_unused_indexes 
WHERE object_schema = 'carpark';
```

3. **索引大小监控**
```sql
SELECT 
    TABLE_NAME,
    INDEX_NAME,
    ROUND(STAT_VALUE * @@innodb_page_size / 1024 / 1024, 2) AS size_mb
FROM mysql.innodb_index_stats
WHERE database_name = 'carpark'
  AND TABLE_NAME LIKE '%vehicle%'
ORDER BY size_mb DESC;
```

### 连接池注意事项

1. **连接数配置**
   - 根据实际并发用户数调整
   - 不要超过MySQL的max_connections

2. **连接泄漏检测**
   - 确保每次查询后都关闭连接
   - 使用try-finally确保资源释放

3. **监控连接池状态**

## 📞 问题排查

### 索引未生效

**检查：**
```sql
EXPLAIN SELECT ... FROM table WHERE ...;
```

**可能原因：**
- 索引名称错误
- 查询条件不匹配索引
- MySQL优化器选择了其他执行计划

**解决：**
```sql
-- 强制使用索引
SELECT ... FROM table FORCE INDEX (idx_name) WHERE ...;
```

### 连接池耗尽

**症状：**
- 报错：`PoolError: connection pool exhausted`

**解决：**
- 增加 `maxconnections` 参数
- 检查是否有连接泄漏
- 减少查询超时时间
